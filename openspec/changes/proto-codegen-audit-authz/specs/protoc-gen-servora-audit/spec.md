## ADDED Requirements

### Requirement: protoc-gen-servora-audit reads audit_rule method option

`cmd/protoc-gen-servora-audit` SHALL be a protoc plugin that scans all `.proto` files marked for generation, extracts `servora.audit.v1.audit_rule` method options from RPC methods, and produces Go source files.

#### Scenario: Plugin discovers annotated RPC

- **WHEN** a proto file contains a service with an RPC annotated with `option (servora.audit.v1.audit_rule) = { enabled: true, event_type: AUDIT_EVENT_TYPE_RESOURCE_MUTATION, mutation_type: RESOURCE_MUTATION_TYPE_CREATE, target_type: "greeting" }`
- **THEN** the plugin SHALL include this RPC in its rule output

#### Scenario: Plugin skips unannotated RPC

- **WHEN** a proto file contains an RPC without an `audit_rule` option
- **THEN** the plugin SHALL not generate any audit rule for that RPC

#### Scenario: Plugin skips disabled rules

- **WHEN** an RPC has `option (servora.audit.v1.audit_rule) = { enabled: false }`
- **THEN** the plugin SHALL not generate an audit rule for that RPC

#### Scenario: No annotated RPCs produces no output

- **WHEN** no RPC in any generated proto file has an `audit_rule` annotation
- **THEN** the plugin SHALL not generate any file

### Requirement: Generated output uses func returning copy

The plugin SHALL generate an `audit_rules.gen.go` file per proto package containing annotated RPCs. The file SHALL contain:
- An unexported `var _auditRules` holding the rule map
- An exported `func AuditRules() map[string]audit.Rule` that returns a shallow copy of `_auditRules`
- The file header `// Code generated by protoc-gen-servora-audit. DO NOT EDIT.`

#### Scenario: Generated func returns independent copy

- **WHEN** a caller invokes `AuditRules()` twice and modifies one copy
- **THEN** the other copy and the internal `_auditRules` SHALL remain unchanged

#### Scenario: Generated file uses correct package name

- **WHEN** the proto package generates Go code in package `sayhellopb`
- **THEN** the generated file SHALL declare `package sayhellopb`

### Requirement: Generated rules map operation to audit.Rule

Each rule entry SHALL be keyed by the full gRPC method path (`/<proto.package.ServiceName>/<MethodName>`) and SHALL contain:
- `EventType` mapped from the proto `AuditEventType` enum value to `audit.EventType` constant
- `MutationType` mapped from the proto `ResourceMutationType` enum value to `audit.ResourceMutationType` constant
- `TargetType` from the proto `target_type` string
- `RecordOnError` from the proto `record_on_error` bool

#### Scenario: Full method path as key

- **WHEN** the proto has `package servora.sayhello.service.v1` and service `SayHelloService` with RPC `Hello`
- **THEN** the rule key SHALL be `"/servora.sayhello.service.v1.SayHelloService/Hello"`

#### Scenario: Enum mapping for mutation_type

- **WHEN** the annotation has `mutation_type: RESOURCE_MUTATION_TYPE_CREATE`
- **THEN** the generated rule SHALL have `MutationType: audit.ResourceMutationCreate`

### Requirement: Generated rules include TargetIDFunc for target_id_field

When `target_id_field` is specified in the annotation, the plugin SHALL generate a field extraction function and assign it to `Rule.TargetIDFunc`.

#### Scenario: Extract from request field

- **WHEN** the annotation has `target_id_field: "req.id"`
- **THEN** the plugin SHALL generate a function that casts `req` to the RPC request type and calls `GetId()`

#### Scenario: Extract from response field

- **WHEN** the annotation has `target_id_field: "resp.id"`
- **THEN** the plugin SHALL generate a function that casts `resp` to the RPC response type and calls `GetId()`

#### Scenario: No target_id_field omits TargetIDFunc

- **WHEN** the annotation does not specify `target_id_field`
- **THEN** the generated rule SHALL not set `TargetIDFunc` (nil)

#### Scenario: TargetIDFunc handles nil safely

- **WHEN** the extraction function receives a nil request or response
- **THEN** it SHALL return an empty string without panicking

### Requirement: Service alias expansion

The plugin SHALL discover all proto packages defining a service with the same short name (e.g. `SayHelloService` in both `servora.sayhello.service.v1` and `servora.sayhello.v1`) and generate rules for all operation path aliases, consistent with `protoc-gen-servora-authz` behavior.

#### Scenario: Single service no alias

- **WHEN** a service `SayHelloService` exists only in `servora.sayhello.service.v1`
- **THEN** rules SHALL use only the `servora.sayhello.service.v1.SayHelloService` prefix

#### Scenario: Service with multiple package aliases

- **WHEN** `SayHelloService` exists in two packages
- **THEN** rules SHALL be generated for both full operation paths
