## Context

Servora Phase 1–2b 已建立完整的审计链路（proto → emitter → Kafka → audit service → ClickHouse → 查询 API）。当前 `resource.mutation` 类审计事件仍需在各服务的 `grpc.go` 中手写 `audit.WithRules(map[string]audit.Rule{...})`，且 `protoc-gen-servora-authz` 生成的 `AuthzRules` 是可变全局 `var`。

Phase 3 的核心是实现 **all-in-proto 声明式审计**，同时改进 authz 插件的安全性。

现有基础设施：


## Goals / Non-Goals

**Goals:**

1. 实现 `protoc-gen-servora-audit`：从 proto `audit_rule` 注解生成 `audit_rules.gen.go`
2. 生成产物包含 `target_id_field` 运行时提取逻辑
3. `annotations.proto` 中 `operation` → `ResourceMutationType mutation_type`（复用已有枚举）
4. `protoc-gen-servora-authz` 输出从 `var` 改为 `func()`（不可变，copy 返回）
5. `pkg/audit` 和 `pkg/authz` middleware 适配 func 签名
6. 以 sayhello 为验证：添加注解 → 生成 → 替换手写 → E2E 验证
7. 集成到 `make api` 生成链路
8. `pkg/` 框架包去特化：清理 actor/authz/scope middleware 中的业务硬编码

**Non-Goals:**

- 不处理 `authn.result` 事件生成（Phase 4）
- 不修改 Audit Service 消费/存储
- 不做运行时动态规则热加载
- 不改造 sayhello 为分层完备微服务
- 不修改 `pkg/authn`（已正确 option 化）
- 不修改 `pkg/transport/server/middleware/identity.go`（已正确 option 化）

## Decisions

### D1: 生成函数签名 — `func AuditRules()` 而非 `var AuditRules`

**选择**: 两个插件统一使用 `func XxxRules() map[string]xxx.Rule` 返回 copy。

**理由**: `var` 全局 map 可被调用方运行时修改，破坏不可变性。`func` 返回 copy 确保：
- 并发安全（每次调用返回独立 map）
- 不可变（原始数据在 unexported `var` 中）
- 与 codegen "不允许更改" 的语义一致

**代价**: 每次调用有 map copy 开销，但规则数量通常 <100，开销可忽略。

**生成代码形式**:
```go
// Code generated by protoc-gen-servora-audit. DO NOT EDIT.
package sayhellopb

import audit "github.com/Servora-Kit/servora/pkg/audit"

var _auditRules = map[string]audit.Rule{
    "/servora.sayhello.service.v1.SayHelloService/Hello": {
        EventType:     audit.EventTypeResourceMutation,
        MutationType:  audit.ResourceMutationCreate,
        TargetType:    "greeting",
        RecordOnError: true,
        TargetIDFunc:  _extractHelloTargetID,
    },
}

func AuditRules() map[string]audit.Rule {
    m := make(map[string]audit.Rule, len(_auditRules))
    for k, v := range _auditRules {
        m[k] = v
    }
    return m
}
```

### D2: `operation` → `ResourceMutationType mutation_type`

**选择**: 将 `annotations.proto` 中 `string operation = 3` 替换为 `ResourceMutationType mutation_type = 3`。

**理由**: CRUD 语义是固定的（create/update/delete），使用枚举而非自由字符串：
- 编译期校验，杜绝拼写错误
- 复用 `audit.proto` 已定义的 `ResourceMutationType` 枚举
- 与 `ResourceMutationDetail.mutation_type` 类型一致

**Proto 变更**:
```protobuf
message AuditRule {
  bool enabled = 1;
  AuditEventType event_type = 2;
  ResourceMutationType mutation_type = 3;  // was: string operation
  string target_type = 4;
  string target_id_field = 5;
  bool record_on_error = 6;
}
```

### D3: `target_id_field` 代码生成策略

**选择**: 在 codegen 阶段为每个带 `target_id_field` 的规则生成一个 field path 提取函数。

**方案对比**:

| 方案 | 优点 | 缺点 |
|------|------|------|
| A. 运行时 reflect | 零 codegen 复杂度 | 运行时 panic 风险、无编译期检查 |
| B. codegen 生成提取函数 | 编译期验证字段存在性、类型安全 | codegen 复杂度高 |
| C. 不提取 ID | 简单 | 审计事件缺少 target ID，丧失精确可审计性 |

**选择方案 B**: codegen 为每个含 `target_id_field` 的规则生成提取函数。

**`target_id_field` 语法**:
- `"req.id"` — 从请求 message 的 `id` 字段提取
- `"resp.id"` — 从响应 message 的 `id` 字段提取
- `"req.user.id"` — 嵌套路径

**生成代码形式**:
```go
func _extractHelloTargetID(req, resp any) string {
    if r, ok := resp.(*HelloResponse); ok && r != nil {
        return r.GetReply()
    }
    return ""
}
```

**Runtime 适配**: `audit.Rule` 新增 `TargetIDFunc func(req, resp any) string` 字段。middleware 在 handler 返回后调用此函数填充 `TargetInfo.ID`。

### D4: 生成文件输出 — per-service package

**选择**: 每个含 `audit_rule` 注解的 proto package 生成独立的 `audit_rules.gen.go`，与 authz 插件行为一致。

**理由**: 各服务的审计规则归属于各自的 proto package，生成文件的 import path 自然跟随 proto package，方便服务引用：`sayhellopb.AuditRules()`。

### D5: Authz 插件同步改造

**选择**: 本阶段同步将 `protoc-gen-servora-authz` 的输出从 `var AuthzRules` 改为 `func AuthzRules()`。

**Breaking Change 迁移**:
1. 修改 `cmd/protoc-gen-servora-authz/main.go` 中的 `generateFile` 函数
2. `make api` 重新生成所有 `authz_rules.gen.go`
3. 调用方（`app/iam/service/internal/server/grpc.go`、`http.go`）将 `iampb.AuthzRules` → `iampb.AuthzRules()`
4. `pkg/authz/authz.go` 中 `WithAuthzRules` 同时支持 `map[string]AuthzRule`，新增 `WithAuthzRulesFunc(func() map[string]AuthzRule)` 或直接调整签名

**推荐做法**: `WithAuthzRules` 直接改为接受 `func() map[string]AuthzRule`，因为代码生成和手写都可以轻松适配此签名。

### D6: buf.audit.gen.yaml 与 make api 集成

```yaml
# buf.audit.gen.yaml
version: v2
managed:
  enabled: true
plugins:
  - local: protoc-gen-servora-audit
    out: api/gen/go
    opt: paths=source_relative
```

`Makefile` 中 `make api` 已有 `buf.go.gen.yaml` + `buf.authz.gen.yaml`，新增 `buf.audit.gen.yaml` 为第三步。

### D7: sayhello proto 注解示例

```protobuf
import "servora/audit/v1/annotations.proto";

service SayHelloService {
  rpc Hello(HelloRequest) returns (HelloResponse) {
    option (servora.audit.v1.audit_rule) = {
      enabled: true
      event_type: AUDIT_EVENT_TYPE_RESOURCE_MUTATION
      mutation_type: RESOURCE_MUTATION_TYPE_CREATE
      target_type: "greeting"
      record_on_error: true
    };
  }
}
```

### D8: pkg/actor 去特化 — 删除业务 scope 常量与便捷方法

**选择**: 删除 `ScopeKeyTenantID` / `ScopeKeyOrganizationID` / `ScopeKeyProjectID` 常量、6 个便捷方法（`TenantID()` / `SetTenantID()` 等）、3 个 context helper（`TenantIDFromContext` 等）。

**理由**: 这些是 IAM 业务领域概念。`Scope(key string)` 和 `SetScope(key, val string)` 已经是通用 API，调用方按需定义自己的 key 常量即可。

**新增**:
- 通用 `ScopeFromContext(ctx context.Context, key string) (string, bool)` — 替代 3 个具名 helper

**迁移**: `app/iam/service` 等调用方将 `actor.ScopeKeyTenantID` 改为自行定义的常量，`actor.TenantIDFromContext(ctx)` 改为 `actor.ScopeFromContext(ctx, "tenant_id")`。

### D9: pkg/actor 清理 legacy Metadata

**选择**: 删除 `UserActorParams.Metadata`、`UserActor.metadata` 字段、`Metadata()` 和 `Meta()` 方法。

**理由**: 全局搜索无任何 `pkg/` 或 `app/` 代码调用 `Metadata()` 或 `Meta()`，为 dead code。`Attrs() map[string]string` 已覆盖同等功能。

### D10: pkg/actor/system.go — ID 前缀去硬编码

**选择**: `SystemActor.ID()` 不再自动拼接 `"system:" + serviceName`，改为构造方提供完整 ID。

**理由**: `"system:"` 前缀是 OpenFGA tuple 语法约定，不应嵌入 actor 包。调用方在构造时传入 `"system:my-service"` 或由 middleware 组装。

**变更**:
```go
// Before
func (s *SystemActor) ID() string { return "system:" + s.serviceName }

// After
func (s *SystemActor) ID() string { return s.id }
```

`NewSystemActor` 签名保持 `NewSystemActor(id string)`（去掉 `serviceName` 概念，或同时保留 `ServiceName()` 但 ID 由调用方控制）。

### D11: pkg/transport/server/middleware/scope.go — 可配置化

**选择**: `ScopeFromHeaders()` 改为接收配置，不再硬编码 3 个 header。

**新 API**:
```go
type ScopeBinding struct {
    Header   string            // HTTP header name (e.g. "X-Tenant-ID")
    ScopeKey string            // actor scope key (e.g. "tenant_id")
    Validate func(string) error // optional validator (e.g. uuid.Parse)
}

func ScopeFromHeaders(bindings ...ScopeBinding) middleware.Middleware
```

**迁移**: 调用方从 `ScopeFromHeaders()` 改为 `ScopeFromHeaders(tenantBinding, orgBinding, projBinding)`，自行定义 bindings。

### D12: pkg/authz — principal 构造泛化

**选择**: `pkg/authz/authz.go` 中将 `"user:" + userID` 硬编码改为根据 `actor.Type()` 动态构造 OpenFGA principal。

**映射规则**:
```go
func principalFromActor(a actor.Actor) string {
    return string(a.Type()) + ":" + a.ID()
}
```

**附带改动**:
- 移除 `a.Type() != actor.TypeUser` 硬判断 — service actor 也应能通过 authz Check
- `"default"` fallback ID 保留为默认值，但通过 `WithDefaultObjectID(id string)` option 可覆盖

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| codegen 为 `target_id_field` 生成提取函数增加插件复杂度 | 先支持简单路径（`req.field`/`resp.field`），嵌套路径作为增量 |
| `var` → `func()` 是 BREAKING change，影响所有已有调用方 | 调用方有限（IAM server grpc/http），全量改动范围可控 |
| proto `operation` → `mutation_type` 变更导致已生成代码不兼容 | `make api` 一次性重新生成，无运行时旧数据存在 |
| sayhello 不是分层完备微服务，可能遗漏某些集成场景 | sayhello 仅作为 codegen 验证目标，分层完备服务的集成模式在文档中说明 |
| 插件编译需要依赖 `api/gen/go` 中的 proto 生成代码 | 与 authz 插件相同的模式，先 `buf generate` 后编译插件 |
| actor/scope/authz 去特化是 BREAKING change，影响 IAM 等调用方 | IAM 已废弃，仅保留编译通过即可；scope 去特化影响的是手写代码，不影响 proto 生成 |
| SystemActor ID 去前缀可能影响 OpenFGA tuple 构造 | 调用方显式传入完整 ID（如 `"system:my-svc"`），语义更清晰 |

## Migration Plan

1. **pkg 去特化（先行）**: actor scope 清理 → scope middleware 可配置化 → authz principal 泛化 → dead code 清理
2. **适配调用方**: IAM/sayhello 适配 actor/scope/authz 变更，确认编译通过
3. **Proto 变更**: 修改 `annotations.proto`（`operation` → `mutation_type`）
4. **重新生成 proto**: `make api`
5. **改造 authz 插件**: `var` → `func()`，重新生成 `authz_rules.gen.go`
6. **适配调用方**: IAM server `AuthzRules` → `AuthzRules()`
7. **新建 audit 插件**: 实现 `cmd/protoc-gen-servora-audit`
8. **新增 buf.audit.gen.yaml**: 配置审计生成模板
9. **sayhello 验证**: 添加注解 → `make api` → 验证生成产物 → 移除手写 rules
10. **更新 pkg/audit middleware**: 适配 `TargetIDFunc` 和 func 签名
11. **E2E 验证**: `make compose.dev` → 调用 sayhello → 检查审计事件是否落库

**回滚策略**: 各步骤独立可回退。最坏情况回退到手写 `WithRules` + `var AuthzRules`。
